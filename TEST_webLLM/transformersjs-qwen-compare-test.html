Standalone harness for @huggingface/transformers (Transformers.js) with ONNX Runtime Web. For model freshness and easier onboarding of new ONNX models (versus MLC’s compile pipeline), this stack usually fits browser-only use well: it runs fully in the browser, can use WebGPU, and loads models from the Hub like ordinary Transformers ONNX exports. Hugging Face documents that Transformers.js runs models in the browser with no server and uses ONNX Runtime underneath; ONNX Runtime Web documents browser deployment with JavaScript APIs. Defaults use small Qwen2.5-family ONNX repos closest to the WebLLM page’s two sizes: onnx-community/Qwen2.5-0.5B-Instruct vs onnx-community/Qwen2.5-1.5B-Instruct. Note: very new exports (e.g. Qwen3.5 / multimodal tokenizers) may list locally but still fail at generation until Transformers.js adds support — use the built-in Qwen2.5 ids to confirm the page works. Each field is a dropdown of curated Hub model_ids: local disk entries live under transformersjs-assets/ at the site root — same parallel layout as webllm-assets/ for the WebLLM page: Hugging Face repo tree per model_id, e.g. transformersjs-assets/onnx-community/Qwen2.5-0.5B-Instruct/. Symlink transformersjs-assets in the repo to the folder where you keep those ONNX exports. Optional transformersjs-assets/_catalog.json (see scripts/refresh-onnx-transformers-catalog.mjs) merges every downloaded model into the dropdown. Large models may fail on limited GPU/RAM — errors are caught and shown. On load the page checks transformersjs-assets/<model_id>/config.json on this origin. If missing (e.g. GitHub Pages), weights come from the Hub. Status lines show which path was used.
